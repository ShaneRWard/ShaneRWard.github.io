GitHub & Git

Workflow

Workflow is the process of uploading your files from your computer, the local repository, to GitHub, the remote repository. It's a fairly simple process done through the terminal. If you're not familiar with the command line interface or Git, check out their respective pages to learn more.

All the files within your repository are either tracked or untracked. This is also called staged or unstaged. Any time something changes, whether it's a file being updated or added, that change is noted. New files are untracked until they are added which we will cover in the next couple steps.

Status

The first part of the process is actually optional. You can check to see what tracked files have been changed or added to the repository. The only exception to that is if said file is listed in the .gitignore file. To do this, open up the git terminal and enter the following command: git status. It will show you something like the following:

Workflow.html is an untracked file. All untracked files and all files that have been changed appear in red. This file needs to be added. For this we will proceed to the next step.

Add

In this step, we will be adding a file to be tracked. Let's go to the terminal and enter a new command to do this. There are actually a couple different ways this can be done. The first is "git add (filename)" to add a specific file. If you have multiple files, this may be a little tedious. The second way is to add all files that are untracked or changed. Enter "git add --all" or alternatively "git add *". Now, if we do another git status we will see the following:

The file is now being tracked and any changes done are ready to be committed. This is indicated by the file name in green text. This means we can now commit the file.

Commit

The next step is to Commit the file to the repository. You will need to enter "git commit -m"(commit message)". The -m"" part is to include a message indicating what the commit does. With the example below, it correctly suggests that the commit "Adds example.html" to the repository. While technically not required to commit the file, it's important and a good practice to add these messages.

Once it's committed, if we do a git status once again, we get what you see below. It shows that you are ahead of the master by 1 commit, meaning that the master is not yet up to date, and the working tree is clean.

This file must be pushed to the origin/master and published to GitHub to have everything up to date. That said, there's only one thing left to do from here. Let's proceed on to the the next and final step.

Push

The final step is to push the file to the remote repository. This publishes you local commit to the origin or master branch on GitHub. In the terminal, enter "git push". You will then be asked for your login credintials. Assuming you've entered your credintials correctly, your file will then be sent to the remote repository.

So, with that the process is complete. The file has been published to the origin/master. If we do one last git status, we see that everything is up to date, there's nothing left to commit, and the workiing tree is clean.

When we go onto GitHub and look at our files, we see example.html among the other files in our repository. This shows we have successfully pushed and published it. This doesn't necessarily mean you are done working on example.html. Perhaps you have more to add or some tweeking to do. Those changes will be tracked and you can do the process over again to publish those changes.
