Semantic Versioning

You may not have ever heard of samantic versioning (SemVer) before, but chances are you have seen it at one point or another in the last several decades. It's an important system that helps to keep track of changes that happen with software. SemVer is esssentially a way of numbering versions of software. It's a three number system where each number tells something about the version and the numbers are each separated by a period. If ever you've loaded up an application on your phone or a program on your computer, there was likely a version number shown somewhere in the window. If you go into the "about" menu or perhaps in the "help" menu in a program, it will display the current version of that software using semantic versioning. Take the image below for example. This is from the "About" section in Visual Studio Code. It shows that the current version in use is 1.30.2.

So, what do these numbers mean? There's a few things to note about these numbers before we explain what each of them means. A piece of software needs to have a publicly declared API. There needs to be a starting point, a public release from which people can measure and track the progression of updates. The first publicly available version should have the number 1.0.0. Prior to release, while it's still in initial development, one could still keep track of the version, but those should start at 0.1.0. One may also use pre-release strings, such as alpha1 for example. These are ammended to the end of the semantic versioning number.

Reading it from left to right, the first number indicates the major version number, the middle indicates a minor version, and the last indicates a patch number. Whenever the major version changes, it breaks from the API of the previous version and there will be no backward compatibility to it. The minor version indicates a new feature or big changes. It doesn't break the backwards compatibility or the API. So, in the case of our version of Visual Studio code shown above, it's still compatible with version 1.29.0. The last number, the patch number, is used to fix bugs, very minor updates to fix small problems. If you have to break backwards compatibility to fix bugs, you will have to do a whole new major version change.

So, looking again at our VS Code version. There have not been any major version changes since its release, but there have been 30 minor changes and feature updates, and 2 bug fixes.

Why use semantic versioning? There are a few benefits that make the whole process easier and cleaner. It makes compatibility or dependency issues easier to follow. It encourages well defined APIs and it makes it rather clear how to label versions when new upgrades, updates, and fixes are made.

More information can be found at http://semver.org.
